## 二、数据模型

### 2.1 Post 相关（apps/web/prisma/schema.prisma）

```
Post (id, userId, content, eventId?, location?, viewCount, likeCount, commentCount, isVisible, createdAt, updatedAt)
  ├── PostImage (postId, imageUrl, width, height, order)
  ├── PostLike (postId, userId, createdAt) @@unique([postId, userId])
  ├── PostFavorite (postId, userId)        @@unique([postId, userId])
  ├── PostView (postId, userId?, ipAddress)@@unique([postId, userId])
  ├── PostReport (postId, userId, reason, status)
  └── Comment (postId, userId, content, parentId?, likeCount)
```

要点：
- `likeCount`、`commentCount`、`viewCount` 是反范式字段，在点赞/评论/浏览时由 API 路由维护。
- `isVisible` 为软删除/封禁标志，所有列表查询都需带上 `where.isVisible = true`。
- `eventId` 可为空——帖子可独立存在或绑定到具体活动。

### 2.2 用户与关注

```
User (id, email?, phone?, nickname, avatar, bio, coverImage, website, location,
      isVerified, verifiedType, followerCount, followingCount, role, isBanned, ...)

UserFollow (followerId, followingId, createdAt) @@unique([followerId, followingId])
```

要点：
- `followerCount`/`followingCount` 是反范式计数，在 follow/unfollow API 内事务维护。
- `verifiedType ∈ {celebrity, artist, organizer, official}`，认证申请走 `VerificationRequest` 表（管理员审核）。

### 2.3 聊天

```
Conversation (id, type[private|group], name?, avatar?, creatorId?, maxMembers, memberCount, lastMessageAt)
ConversationParticipant (conversationId, userId, role[owner|admin|member], unreadCount, lastReadAt)
                                                                          @@unique([conversationId, userId])
Message (id, conversationId, senderId, receiverId?, content, messageType, isRead, readBy?)
```

要点：
- `Conversation.type` 区分私聊与群聊；私聊也走 Conversation 表（不直接 user-to-user）。
- `Message.receiverId` 仅私聊使用；群聊为空，已读用 JSON 数组 `readBy` 记录。
- `unreadCount` 在 ConversationParticipant 上而非 Conversation 上，每个成员独立计数。

## 三、API 一览

### 3.1 帖子（Post）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/posts` | 列表，支持 `page` `pageSize` `eventId` `userId` `sort=latest\|hot\|following` |
| POST | `/api/posts` | 发帖（认证），含图片数组 |
| GET | `/api/posts/[id]` | 详情 |
| PUT/DELETE | `/api/posts/[id]` | 编辑 / 删除（仅作者） |
| POST/DELETE/GET | `/api/posts/[id]/like` | 点赞 / 取消 / 状态 |
| POST/DELETE/GET | `/api/posts/[id]/favorite` | 收藏 / 取消 / 状态 |
| POST | `/api/posts/[id]/report` | 举报 |
| GET/POST | `/api/posts/[id]/comments` | 评论列表 / 发评论 |
| DELETE | `/api/posts/[id]/comments/[commentId]` | 删除评论 |
| POST/DELETE | `/api/posts/[id]/comments/[commentId]/like` | 评论点赞 |

`sort` 参数语义（2026-04-28 新增）：
- `latest` （默认）：按 `createdAt desc`
- `hot`：仅近 7 天，按 `[likeCount desc, commentCount desc, createdAt desc]`
- `following`：仅当前用户关注的人发的帖子；未登录或未关注任何人时返回空数组

### 3.2 用户与关注

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/user/me` | 当前用户资料 |
| PUT | `/api/user/me` | 更新当前用户资料 |
| GET | `/api/user/[id]` | 任意用户公开资料（含 `isFollowing`、`isFollowedBy`、stats） |
| GET | `/api/user/[id]/following` | 该用户的关注列表 |
| GET | `/api/user/[id]/followers` | 该用户的粉丝列表 |
| GET/POST/DELETE | `/api/users/[id]/follow` | 关注状态 / 关注 / 取消关注（注意是 `users` 复数） |
| GET | `/api/users/search` | 搜索用户 |
| GET | `/api/user/follows` | 当前用户关注的**活动**（不是用户，命名易混淆） |
| GET | `/api/user/following` | 当前用户关注的**用户**列表（旧版接口，仍在使用） |

⚠️ 命名陷阱：
- `/api/user/follows` —— 关注的**活动**（EventFollow 表）
- `/api/users/[id]/follow` —— 关注**用户**操作（UserFollow 表）
- 这是历史命名遗留，新代码统一使用后者。Mobile 端历史 bug 误调前者已于 2026-04-28 修复。

## 四、实时通信架构

### 4.1 服务端（apps/web/server.js）

Next.js 通过自定义服务器（`node server.js`）启动，附带 Socket.IO Server：

- `httpServer = createServer(handle)` —— 包裹 Next 的 request handler
- `new Server(httpServer, { path: '/socket.io/' })` —— 同端口启动 socket.io
- `global.io = io` —— 让 API 路由通过 `lib/socket.ts` 的 `emitToUser/emitToUsers/broadcast` 主动推送

**鉴权**：connect 时通过 `socket.handshake.auth.token` 传 JWT，`io.use` 中间件 `jwt.verify` 后绑定 `socket.userId`。

**房间**：`socket.join('user:' + userId)` —— 后端通过 `io.to('user:xxx').emit(...)` 即可定向推送。

**事件**：

| 事件 | 方向 | 用途 |
|------|------|------|
| `message:new` | server → client | 新消息推送 |
| `message:sent` | server → client | 发送方确认 |
| `message:read` | both | 已读回执 |
| `typing:start` / `typing:stop` | both | 打字指示 |
| `user:online` / `user:offline` | server → client | 在线状态 |
| `post:join` / `post:leave` | client → server | 进入/离开帖子房间（用于实时评论） |
| `comment:new` | server → client | 新评论推送（帖子房间内） |

**部署**：`apps/web/Dockerfile` 末尾 `CMD ["node", "server.js"]`，生产同样走自定义服务器（不是 `next start`）。

### 4.2 客户端（apps/mobile）

| 文件 | 角色 |
|------|------|
| `src/services/socket.ts` | Socket 单例 `socketService`，封装 connect / emit / on / off |
| `src/contexts/SocketContext.tsx` | 全局 Provider：管理连接生命周期、集中监听事件写入 store、AppState 同步、Polling fallback |
| `src/stores/messagingStore.ts` | Zustand store：会话列表、按会话分组的消息、未读、分页、打字状态 |

数据流：

```
后端 emitToUser → socket.io → SocketContext.handleNewMessage
                                     ↓
                       messagingStore.addMessage（带去重 + optimistic 替换）
                                     ↓
                       页面通过 store 选择器订阅，自动 re-render
```

页面侧不直接监听 socket 事件 —— 所有事件统一由 SocketContext 落地到 store。

## 五、关键数据流

### 5.1 发送消息（Mobile）

```
ChatScreen.handleSendMessage(content)
  ├── messagingStore.addOptimisticMessage(conversationId, content, ...) → 临时 ID `temp_xxx`
  ├── messagingStore.sendMessage(conversationId, content)
  │     └── apiSendMessage → POST /api/messages/conversations/[id]/messages
  │           └── 后端创建 Message → 推送 message:new 给所有参与者（含发送者）
  ├── 真实 message 返回 → replaceOptimisticMessage(tempId, realMessage)
  └── 同时 SocketContext 收到 message:new → addMessage(realMessage)
        └── addMessage 检查：
             ① 真实 ID 已存在 → 跳过
             ② 找到 senderId+content 匹配的 optimistic → 用 realMessage 替换
             ③ 否则 prepend
```

无论 socket 推送和 HTTP 响应哪个先到，最终结果都是「列表里只有一条真实消息」。

### 5.2 关注用户

```
UserProfileScreen.handleFollow
  ├── isFollowing ? unfollowUser(userId) : followUser(userId)
  │     → apiClient (POST/DELETE) /api/users/{userId}/follow
  ├── 后端事务：
  │     ① 创建/删除 UserFollow 记录
  │     ② follower.followingCount ±1
  │     ③ following.followerCount ±1
  └── 前端：本地切换 isFollowing + 更新 followerCount
```

### 5.3 拉取 Feed

```
EncoreScreen.loadPosts
  └── getPosts({ page, limit, sort, userId?, eventId? })
        → GET /api/posts?...
              ├── sort=latest → orderBy: createdAt desc
              ├── sort=hot    → where.createdAt >= 7天前; orderBy: [likeCount, commentCount, createdAt]
              └── sort=following → 取关注列表 ID → where.userId IN (...)
```

## 七、已知问题与最近改动

### 7.1 2026-04-28 修复（已合入）

| # | 问题 | 影响 | 修复 |
|---|------|------|------|
| 1 | Mobile `followUser/unfollowUser` 调用 `/api/user/follows/{id}`，端点不存在 | 关注按钮 100% 失效（未冒烟到的生产 bug） | 改为 `/api/users/{id}/follow` |
| 2 | 消息推送只发给「非发送者」，多端登录时另一台设备收不到自己的消息 | 用户感知"需要刷新"的根因之一 | 改为推送给所有参与者，附 `isSelfEcho` 标志 |
| 2.1 | 上述改动连带影响：单聊 receiverId 从 `participants[0]` 取，可能取到发送者本人 | 数据错误 | 显式 `find(p.userId !== senderId)` |
| 2.2 | 发送者收到 self-echo 后会与本地 optimistic 消息撞车 | 短暂双消息闪烁 | `messagingStore.addMessage` 增加 optimistic 替换逻辑（按 senderId+content 匹配 `temp_*`） |
| 3 | App 切回前台后 Socket 可能漏接事件，但 UI 不知情 | "需要刷新"的根因之二 | `SocketContext` 监听 AppState：背景→活跃时 reconnect + `syncAfterReconnect()` |
| 4 | Socket 长时间断连无任何兜底 | 弱网/移动网络下丢消息 | `SocketContext` 增加 polling fallback：断连 30s 后启动 15s 间隔 HTTP 轮询，重连后即停 |
| 5 | `/api/posts` 不解析 `sort`/`userId` 参数 | 前端 EncoreScreen 的「热门」「关注」Tab 实际效果与「最新」一致 | 后端实现 `hot`（7 天 + 互动量降序）和 `following`（关注列表过滤） |
| 6 | Web 无他人用户主页 | 桌面用户分享链接无法直达 | 新建 `apps/web/app/u/[id]/page.tsx` + `ui/UserProfileClient.tsx` |
| 7 | Mobile 私聊三页缺少毛玻璃/阴影/边框，与 Web 主界面不统一 | 视觉割裂 | 新增 `surfaceGlass` token，应用到 ConversationsScreen、ChatScreen、SelectUserScreen、MessageBubble、MessageInput |

| 项 | 现状 | 是否计划改进 |

### 7.3 命名遗留与坑位（给后人）

- 路由 `/api/user/follows`（关注**活动**）vs `/api/users/[id]/follow`（关注**用户**）—— `user` 单复数 + 是否带 `[id]` 决定语义。新代码勿混用。
- `/api/user/following` 返回 `{ data: { followings: [...], pagination } }`（嵌套）；`/api/user/[id]/following` 返回 `{ data: [...], pagination }`（扁平）。两个接口结构不一致，调用前确认。
- `apps/web/app/encore/[id]/ui/PostDetailClient.tsx` 用 `localStorage.getItem('token')` 取 token；其他地方有的用 cookie。Web 端缺统一 token 访问层。
- Mobile 端发消息**只走 HTTP**，不通过 socket emit `message:send`（虽然 `socket.ts` 有这个方法）。后端的 `socket.on('message:send')` 历史代码也无 DB 写入，是死代码。

## 八、开发指南速查

### 8.1 本地启动

```bash
# Web (含 Socket.IO)
cd apps/web && npm run dev   # 实际跑 node server.js

# Mobile
npm run dev:mobile           # Expo dev server
```

确保 Mobile 的 `EXPO_PUBLIC_API_URL` 指向你本地 web 的 IP:3000，而不是 localhost（真机扫码无法 localhost）。

### 8.2 添加新 socket 事件

1. 后端：在 `apps/web/server.js` 的 `io.on('connection')` 内加 `socket.on('xxx')`，或在 API 路由中调 `emitToUser(userId, 'xxx', payload)`。
2. 客户端：在 `src/services/socket.ts` 的 `SocketEvent` 枚举里加事件名；在 `src/contexts/SocketContext.tsx` 的事件监听 useEffect 里 `socketService.on(SocketEvent.Xxx, ...)` 并把数据写进 store。
3. 不要在页面里直接 `socketService.on()`，统一走 SocketContext，否则 unmount 时容易漏掉 off。

### 8.3 添加新关注派生 Feed

参考 `/api/posts?sort=following` 的实现：
1. 解析参数
2. 拿当前用户的关注 ID 集合
3. 拼到 `where.userId = { in: [...] }`
4. 复用现有的 orderBy / pagination

### 8.4 视觉调整规则

- 颜色不要硬编码，引用 `colors.*`
- 卡片用 `surfaceGlass`，加 `shadows.sm`，圆角 `borderRadius.md` 起
- 与 Web 比对：`cards.css` 的 `.content-card` 是基准
