## 一、为什么做这次改造

线上原始版本存在三个互相牵连的问题：

| 问题 | 现象 | 根因 |
|------|------|------|
| 1. 找回管理员密码麻烦 | Web 端没有"修改密码"页，只能进 Prisma Studio 改 hash | `apps/web/app/account/settings/page.tsx` 一直是占位页（"功能开发中"） |
| 2. 输错几次就被锁死 | 5 次错密码后 15 分钟内任何登录都 429 | `middleware.ts` 只有 IP+路径维度的内存 Map 限流，无账号级退避；进程重启清零、多实例不同步 |
| 3. 多人登录或单人请求多就过载 | 网站卡顿、CPU 单核打满 | `bcryptjs` 是纯 JS 实现 + SALT_ROUNDS=12，每次 `verifyPassword` 在 Node 主线程同步阻塞 ~150ms，并发登录直接卡死事件循环 |

三个问题**一次性解决**：账号级退避计数器顺势启用本就部署却闲置的 Redis，bcrypt 换成 async 原生模块后主线程不再被阻塞，限流误杀也跟着缓解。

## 二、现在的整体设计

```
┌────────────────────────────────────────────────────────────────┐
│  浏览器/App                                                     │
└────────────────────┬───────────────────────────────────────────┘
                     │ POST /api/auth/login
                     ▼
┌────────────────────────────────────────────────────────────────┐
│  middleware.ts                                                  │
│  IP 级粗粒度防扫号（防爬虫/防扫号），30/15min                     │
│  内存 Map（重启清零，单实例够用）                                │
└────────────────────┬───────────────────────────────────────────┘
                     ▼
┌────────────────────────────────────────────────────────────────┐
│  /api/auth/login route.ts                                       │
│                                                                 │
│  1. checkLoginAllowed(account)  → Redis 看锁定状态               │
│     - 锁定中：直接 429 返回剩余等待时间                           │
│  2. 查 DB / 验证密码（@node-rs/bcrypt async，不阻塞主线程）       │
│  3a. 成功 → clearLoginFailures + 创建会话 + fire-and-forget 日志 │
│  3b. 失败 → recordLoginFailure                                  │
│       count<5  → 401 + attemptsLeft                             │
│       count>=5 → 429 + retryAfterSec（指数退避）                 │
└────────────────────┬───────────────────────────────────────────┘
                     ▼
┌────────────────────────────────────────────────────────────────┐
│  Redis（lib/redis.ts 单例）                                      │
│  login:fail:{sha256(account)}  INCR + EXPIRE 24h                │
│  login:lock:{sha256(account)}  SET EX 60/120/240/480/960/1800   │
│  Redis 出错时整条链路降级为"放行"，绝不阻塞合法登录              │
└────────────────────────────────────────────────────────────────┘
```

### 退避策略

- **阈值**：连续失败 5 次开始锁定
- **梯度（秒）**：60 → 120 → 240 → 480 → 960 → 1800（封顶 30 分钟）
- **解锁**：成功登录后 `DEL login:fail:* login:lock:*`
- **失败计数自身 TTL**：24 小时（防止永久累积）
- **账号 key**：`sha256(account)` 截前 32 字符，不存原始手机号/邮箱
- **存在性脱敏**：账号不存在时也走计数（避免攻击者用是否触发 429 来枚举账号）；错误信息一律"账号或密码错误"

### bcrypt hash 兼容性

| 库 | 输出格式 | 互通验证 |
|----|---------|----------|
| bcryptjs（旧） | `$2b$12$...` | 已测试：@node-rs/bcrypt 能校验 ✅ |
| @node-rs/bcrypt（新） | `$2y$12$...` | 已测试：bcryptjs 能校验 ✅ |

**所有现有用户密码 hash 不需要重置**。这是选 @node-rs/bcrypt 而非 argon2/scrypt 的核心理由。

### 3. 账号级退避
- 故意错密码 6 次：
  - 第 1-4 次：401 `账号或密码错误`，响应里 `attemptsLeft` 递减
  - 第 5 次：401 `attemptsLeft: 0`
  - 第 6 次：429 `ACCOUNT_LOCKED`，`retryAfterSec ≈ 60`
- 等 60 秒解锁，再连续错 6 次 → 第 6 次 `retryAfterSec ≈ 120`（验证翻倍）
- 验证清零：
  ```bash
  docker compose exec redis redis-cli -a $REDIS_PASSWORD KEYS 'login:*'
  ```
  能看到 `login:fail:*` 和 `login:lock:*`，TTL 合理
- 输对密码登录成功后再 KEYS → 应该清空

### 4. Redis 持久化（关键改进）
- 错 6 次锁定后立即 `docker compose restart web`
- web 起来后再登录 → 仍然 429（之前内存方案这里会清零）

### 5. 过载缓解
- `docker stats piaociyuan-web` 观察 CPU 不再单核 100% 满
- 用 ab 模拟并发登录：
  ```bash

  ab -n 100 -c 20 -p /tmp/login.json -T application/json https://piaociyuan.com/api/auth/login
  ```

## 六、回滚预案

| 故障 | 影响 | 回滚方案 |
|------|------|---------|
| @node-rs/bcrypt 在 Alpine 装不上 | 镜像构建失败 | `package.json` 把 `@node-rs/bcrypt` 删掉，`lib/auth.ts` 改回 `import bcrypt from 'bcryptjs'`、`bcrypt.compare`/`bcrypt.hash`。现有 hash 兼容，无需改数据 |
| Redis 连不上/挂掉 | 登录退避失效 | 代码已内置降级：`login-throttle.ts` 所有调用包了 try/catch，Redis 出错时返回"放行"。**登录功能不会因 Redis 故障中断** |
| 账号级退避误锁真用户 | 用户投诉 | 临时手动清：`docker compose exec redis redis-cli -a $REDIS_PASSWORD DEL login:fail:<sha256> login:lock:<sha256>`。或全清 `redis-cli -a $REDIS_PASSWORD --scan --pattern 'login:*' \| xargs redis-cli -a $REDIS_PASSWORD DEL` |
| 修改密码 UI 报错 | 用户改不了密码 | API 没动，老路径——SSH 进 ECS 用 Studio 改 hash 仍可用 |
